## Why

The first production Inbox Manager canary created one BathOS Mail task successfully, but replaying the accepted idempotency UUID returned `different request`. The database compares caller-controlled content with fresh task identity, planning date, and order values generated by the MCP tool, so an exact retry can fail after an ambiguous response and cannot satisfy the durable handoff contract.

## What Changes

- Make the atomic Mail-capture service compare only caller-controlled fields when an idempotency UUID already exists
- Return the original task, Mail source, and creation receipt with `already_applied` for an exact retry
- Reject a reused idempotency UUID when any caller-supplied Mail-capture field differs
- Add a database regression that retries with different generated task identity, planning date, and ordering
- Apply the corrected function and repeat the accepted production canary UUID without creating another task

## Capabilities

### New Capabilities

None

### Modified Capabilities

- `personal-tasks-module`: Make specialized Mail capture retry-safe even when server-generated task identity and ordering would differ at retry time

## Impact

- `tasks_create_mail_capture` behavior and its database regression suite
- Mail-capture production canary acceptance evidence
- One forward-only Supabase migration that replaces the existing function without changing its signature or grants
- No table schema, RLS, publication, PowerSync stream, Edge Function, reminder, Mail rule, or general dual-write change
